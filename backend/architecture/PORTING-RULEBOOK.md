# Porting rulebook

[`PORT-READINESS.md`](PORT-READINESS.md) inventories how the backend's shapes map onto a native port of this Python sidecar into the Tauri shell; this rulebook turns that inventory into application-ready rules. It is written for the implementer porting one service at a time against the existing test infrastructure ([`../../TESTING.md`](../../TESTING.md)): the replay corpus, the golden files, and the contract gates are the equivalence judges, and these rules exist so a ported service passes them for the right reasons rather than by accident.

How to use it: find the surfaces the service touches, apply every MECHANICAL rule blind (zero latitude; the rule states exactly what to reproduce and what proves it), treat each JUDGEMENT rule as a named decision to make consciously, check the divergence register before "improving" anything, and close out against the verification obligations that touch the service. Every rule carries an in-repo citation and names the oracle that grades it; where a surface is honestly unpinned, the rule says so instead of implying coverage that does not exist.

## The governing split

Three layers govern every porting decision:

- **BOUNDARY**: every externally observable surface (wire bytes, persisted database rows, event frames, on-disk file formats, status codes, header semantics) is reproduced faithfully, byte-for-byte after the corpus normalisation described in rule G-1. Zero latitude. When in doubt whether something is observable, treat it as boundary until shown otherwise.
- **INTERIOR**: implementation internals (lock layouts, task shapes, data structures, library choices) are idiomatic Rust. Latitude is allowed exactly where no oracle observes.
- **DIVERGENCE**: a short register of deliberate structural improvements the port makes by construction (section "Deliberate-divergence register"). Each entry names the change, the reason, and the oracle asset that still proves behavioural equivalence despite the change. Anything not in the register is not a sanctioned divergence.

The split resolves the classic porting tension: fidelity governs the boundary, idiom governs the interior, and the register is the only bridge between them.

- the database write tables of the codex, quest, skill, and equipment services (absent from the `db_snapshot.py` catalogue; observed only through their HTTP read projections and unit suites; where the read body is itself schema-only (analytics, character, equipment), the two gaps compound and the write tables rest on the unit suites alone);
- CORS response headers and the origin/host-guard 403 bodies (deliberately dropped from the HTTP-fingerprint projection);
- the exact 4xx detail strings across routers (the corpus exercises 200 paths);
- keystroke-to-event timing (the keystroke log is replay input, not a golden);
- connection pragma state, the schema-version metadata row, and the in-place upgrade ladder (the corpus replays from schema head);
- the live wire's raw byte form: key order, separators, sub-4dp float rendering (rule W-7 explains why the goldens mask it and what still observes it).

## Coverage as certification

An equivalence oracle is only as strong as its code-path coverage: a Python branch no scenario exercises is a place the port can diverge undetected, with every golden still green. A byte-stable golden certifies the outputs of the paths that ran; it does not certify that the scenarios exercise the service's branches, and a happy-path golden over a branchy service leaves most of the service unobserved.

Two consequences, both MECHANICAL:

1. **Author scenarios before pinning goldens on a thin surface.** Where a surface's branch coverage is thin (the router/API layer is the known concentration), write coverage-guided synthetic scenarios first, using the scenario DSL (`backend/testing/AUTHORING.md`), so the branches are driven into the goldens; only then pin. Pinning whatever scenarios happen to exist certifies less than it appears to.
2. **Measure branch coverage on the ported side; never assume it.** Measure the ported implementation's branch coverage with its own coverage tooling and compare it against the same module's measured Python figure (measure the Python side with the standard coverage tooling where no recorded figure exists); treat any shortfall as a gap to close or to justify explicitly, never as noise. A port whose tests exercise fewer branches than the original's is resting its equivalence claim on paths it never runs.

Relatedly: line coverage proves a path ran; only the mutation score proves the tests would notice a change there (`backend/testing/COVERAGE.md`). The nightly per-module mutation floors are the per-surface answer to "how strong must the ported tests be"; a ported module's mutation score belongs at or above its Python floor.

## The rules

Tags: **[M]** = MECHANICAL (apply blind), **[J]** = JUDGEMENT (named call). Layers: **B** = boundary, **I** = interior, **D** = divergence (see register). A MECHANICAL rule pins the observable with zero latitude; where its *expression* in the port involves a genuine choice, the rule says so explicitly and routes the choice to a named obligation (V-10) rather than leaving silent latitude.

### Wire contract

- **W-1 [M/B] The OpenAPI snapshot is the route surface, byte for byte.** Reproduce every path string literally (`{param}` placeholders included), the `openapi: 3.1.0` envelope, the schema names, and the operationId convention (FastAPI's `<function_name>_<full_path_with_slashes_as_underscores>_<method>`, path params wrapped in double underscores). If the chosen spec generator cannot reproduce these exactly, the snapshot change is a deliberate, ratified golden move plus a client regeneration, never silent drift. Watch route declaration order where a static path shares a prefix with a parameterised one: `backend/routers/quests.py` declares `/mobs`, `/analytics`, `/playlists` before `/{quest_id}` so the static paths win; the ported router table must resolve identically. Pinned by: `openapi.snapshot.json` + `test_openapi_drift.py` + the `gen:api:check` client gate.

- **W-2 [M/B] Key casing is verbatim per field; never apply a global rename policy.** The read surface deliberately mixes casings: the tracking snapshot emits `session_id`, `started_at`, `kill_count` in snake_case beside `returnRate`, `recentEvents`, `weaponCost` in camelCase (`backend/routers/response_models.py`). A blanket camelCase rename on the port's response types silently corrupts the snake_case keys; spell each wire key exactly as the snapshot records it. How to express that in the port (per-field renames, handler-built maps, or a custom serialiser) is a named decision at the wire-layer port (V-10). Pinned by: `openapi.snapshot.json` (the casing lives inside the golden) + the HTTP-response goldens.

- **W-3 [M/B] The uniform-float convention: value fields are floats and render with a decimal point; counts, ranks, and identifiers are integers and render bare.** A contract-float field with an integral value serialises as `0.0`, never `0`; `kills: 3` stays bare. In the port that means value fields are `f64` end to end (Rust's standard JSON serialisation emits `0.0` for an integral `f64`, matching Python's `json.dumps`), and integral fields are integer types. Note that model-typed float fields coerce on the wire: a handler that computes a Python `int` (single-argument `round` returns one) still serialises as `0.0` through a float-typed response-model field, so int-looking zeroes in the HTTP-response goldens are an artefact of the comparison parsing bodies (where `0 == 0.0`), not the live wire form; do not special-case them. Pinned by: the number-vs-integer typing in `openapi.snapshot.json` + schemathesis; the spec and conformance gates, not the body goldens, are the float-vs-int judges.

- **W-4 [M/B] Descriptive response models never truncate handler output.** Every response model is open (`_Loose`, `extra="allow"` in `backend/routers/response_models.py`): a key the handler emits but the model does not declare passes through to the wire. Live examples: `RepairScanResult` gains an undeclared `error` on failure; the character prospect carries undeclared forecast keys. A closed struct that drops undeclared keys is a behavioural break. Either serialise the handler's own value tree, or give each struct an explicit catch-all flatten map. Pinned by: the HTTP-response goldens (full bodies, including the undeclared keys).

- **W-5 [M/B] Absent-vs-null is a per-route property, not a global one.** Routes served with `response_model_exclude_unset=True` (the tracking snapshot, the scan verb responses, the character optimisers, among others; enumerate from the router sources) omit unset keys entirely, which is how the three-state tracking snapshot keeps distinct key sets per status. Other routes deliberately emit explicit `null` for declared optional keys (the scan status when idle, the inventory sell result's `ledgerEntry`). A blanket skip-when-none policy on the port wrongly drops the deliberate nulls; a blanket always-emit policy wrongly fattens the lean shapes. The discriminator to enumerate on is the `response_model_exclude_unset=True` argument on the route decorator: present means unset keys are omitted; absent means declared optional keys the handler sets serialise as explicit `null`. Model the union-shaped responses as proper tagged unions or handler-built maps so absence and null are both expressible. Pinned by: `GET_tracking_snapshot` goldens (idle key set), `test_tracking_snapshot.py`, the `required`/`anyOf` shapes in the OpenAPI snapshot.

- **W-6 [M/B] Error responses are `{"detail": <string>}` at the raised status; reproduce status and shape, and treat detail strings as unpinned until verified.** Business-rule failures map to a JSON object with a single `detail` string at 400/404/409/422/503 per the conventions in each router (`backend/routers/`); request-validation failures produce the framework's `{detail: [...]}` array shape at 422. Both 422 shapes coexist (a string detail from an explicit raise, an array from validation); reproduce both rather than unifying them. The corpus pins 200-path responses only, and the spec documents only 200 and 422 (no router declares additional responses), while the conformance suite checks for the absence of server errors, not status correctness: so the business-rule status codes (400/404/409/503) and every detail string have no oracle today, and the port carries V-6 (sweep and pin them) rather than assuming coverage. Pinned by: 200 via the OpenAPI snapshot and the per-scenario HTTP goldens; 422 via the snapshot's validation responses; the rest: none today (V-6).

- **W-7 [J/B] The live wire's raw bytes are an observable surface even though the goldens mask them.** The goldens re-serialise with sorted keys, so they do not pin the live wire's key order (handler insertion order), separators (compact, no spaces), or sub-4dp float rendering; but the ETag is a SHA-256 of the exact body bytes (M-1), so byte-level serialisation differences surface as ETag churn across implementations even when every golden passes. Reproduce the live wire form: compact separators, insertion-order keys, shortest-round-trip float rendering, UTF-8 without ASCII escaping. The judgement is where to spend verification effort: rule N-8 names the fields where float rendering genuinely bites. Pinned by: nothing end-to-end today (the goldens sentinel the ETag); V-3 makes it a port-side obligation.

### Conditional requests and middleware

- **M-1 [M/B] The ETag is a strong validator: a quoted lowercase SHA-256 hex of the exact response body bytes, with `Cache-Control: no-cache` beside it.** Format `"<64 hex chars>"`, no `W/` prefix, computed over the serialised body after the inner layers produced it; the middleware owns the `Cache-Control` header (any handler-set value is replaced) and its value is the literal `no-cache`. Equal bodies must yield equal ETags wherever they are produced. Pinned by: `test_etag.py` (format regex, stability across calls).

- **M-2 [M/B] Conditional-request scope is exactly four prefixes, 2xx GETs only.** `/api/tracking`, `/api/scan`, `/api/quests`, `/api/codex`, matched as the exact path or prefix-plus-`/` (so `/api/codex2` does not match). Everything else (including `/api/demo/*`, `/api/events`, analytics, character, equipment, settings) carries no ETag and no middleware-set Cache-Control; that exclusion is itself asserted. Pinned by: `test_etag.py` scope tests; `covered_get_routes` in `backend/middleware/etag.py` enumerates the covered surface by name.

- **M-3 [M/B] If-None-Match follows RFC 7232 weak comparison; a match returns 304 with an empty body and preserved headers.** Strip `W/` from both sides, split on commas, honour `*`, match any candidate. The 304 carries the ETag, Cache-Control, and every header the inner stack added to the would-be 200, minus Content-Type (no body to describe) and Content-Length. Pinned by: `test_etag.py` (the comparison classification table and the 304 tests).

- **M-4 [J/B] Layer ordering: the conditional-request layer is outermost, the origin guard next, CORS innermost of the three.** The Python registration order produces this runtime nesting (Starlette wraps last-registered outermost; `backend/main.py`), and it is load-bearing: the ETag layer must see the final body and the inner layers' headers so a 304 still carries them. Mapping that onto the port's middleware composition is a named translation call; get the nesting wrong and the 304 path strips CORS headers. Pinned by: `test_etag.py` (only passes with the ETag layer observing final responses); the note in `backend/middleware/etag.py`.

- **M-5 [M/B] The origin and host guard on `/api/*`: three rules, three distinct 403 bodies.** Host header present but not in the allowed loopback set: 403 `{"detail": "Invalid Host header"}`. Unsafe methods require a present and allowed Origin: 403 `{"detail": "Origin header required"}`. Safe methods reject only a present-but-disallowed Origin: 403 `{"detail": "Invalid Origin header"}`. OPTIONS passes through to CORS untouched. The allowed sets derive from the configured backend port, frontend port, and the Tauri origins (`backend/main.py`). The three detail strings are unpinned by any golden (V-6). Pinned by: the contract-suite harness behaviour; detail bodies: none today.

- **M-6 [M/B] CORS is an explicit allowlist, six methods, one allowed request header.** Allowed origins are the sorted explicit set (never a wildcard), methods GET/POST/PATCH/PUT/DELETE/OPTIONS, allowed request header `content-type` only. CORS response headers are deliberately outside the HTTP-fingerprint projection, so this configuration is verify-on-port (V-5). Pinned by: none (configuration read from `backend/main.py`).

### Events and streaming

- **E-1 [M/B] The domain-event union is internally tagged on `type`, and the tag values are the dotted topics.** `tracking.session.updated` and `scan.status.changed` serve as bus topic, envelope discriminator, and SSE `event:` field identically; never introduce a translation table. The union is closed: unknown tags are errors, and both envelope and payload reject undeclared keys (the Python models set `extra="forbid"`). The frontend relay's dot-to-colon rename is the frontend's job and out of port scope. Pinned by: `event_schemas.snapshot.json` (discriminator mapping, `additionalProperties: false`) + the wire-dict assertions in `test_event_schema_drift.py`.

- **E-2 [M/B] Envelope keys are snake_case, payload keys are camelCase, in declaration order.** The envelope serialises exactly `type`, `event_version`, `occurred_at`, `payload` in that order; `event_version` and `occurred_at` stay snake_case (a whole-struct camelCase rename corrupts them), while payload fields are camelCase (`sessionId`, not `session_id`). `event_version` is an always-present integer (currently 1 for both events). `occurred_at` is required and non-nullable: every emitter supplies a real instant, synthesising one from its injected clock where the domain carries none, so the wire never carries `"occurred_at": null`. The payload's `sessionId` stays nullable with its key always present: a None serialises as `null`, never as an omitted key, so no skip-when-none serialisation there. Pinned by: `test_event_schema_drift.py` exact wire dicts and the schema snapshot's `required` list (key presence and values; dict comparison does not pin order); the serialisation order and the null-key behaviour are code-sourced and unpinned (V-4).

- **E-3 [M/B] `occurred_at` reproduces Python's `datetime.isoformat()` for a UTC instant: `+00:00` offset, fractional seconds omitted at zero microseconds and rendered as exactly six digits otherwise.** Examples: `2024-12-31T21:20:00+00:00`, `2024-12-31T21:20:00.500000+00:00`. Stock RFC 3339 formatters (trailing `Z`, adaptive precision) do not match; write the formatter to these rules. The input is a float epoch that Python first rounds to the nearest microsecond (ties to even, whole-second carry included) before the omit-at-zero rule applies. For the tracking event the value derives from the domain timestamp (the tick instant, the session boundary), falling back to the injected clock only when a settled tick carries no instant; the scan event stamps its publish instant through the injected clock, and no golden observes its value (T-1). Pinned by: the integer-second case in `test_event_schema_drift.py`; the fractional case is unpinned (V-4).

- **E-4 [M/B] Payload enums serialise to exact lowercase tokens.** `status` in {`active`, `idle`}; `reason` in {`started`, `updated`, `stopped`}; `phase` in {`idle`, `capturing`, `processing`, `awaiting_review`}. Pinned by: the enum arrays in `event_schemas.snapshot.json`.

- **E-5 [M/B] Event JSON is compact.** No whitespace after `:` or `,` anywhere an envelope is serialised to the wire (the SSE `data:` line). Pinned by: substring assertions in `test_event_stream.py`.

- **E-6 [M/B] The SSE contract, byte for byte.** On connect, the opening comment `: ready\n\n` (it flushes headers and signals hub registration; tests synchronise on it). Data frames are exactly `id: <seq>\nevent: <topic>\ndata: <compact envelope JSON>\n\n` (field name, colon, single space; bare LF; blank-line terminator). A quiet stream emits `: keep-alive\n\n` every 15 seconds, which doubles as the disconnect check. Response: status 200, media type `text/event-stream`, headers `Cache-Control: no-cache`, `Connection: keep-alive`, `X-Accel-Buffering: no`. The endpoint stays outside the OpenAPI document and outside the conditional-request layer (an unbounded stream cannot be buffered or hashed). Pinned by: `test_event_stream.py` (frame shape), `test_event_stream_seam.py` (ready comment, content type, live delivery); the keep-alive text, cadence, and two of the three headers are unpinned (V-4).

- **E-7 [M/B] Fan-out: per-connection bounded delivery, capacity 256, drop-oldest; one process-monotonic sequence starting at 1; no replay.** The `id:` value is a single counter shared across connections, assigned at the one fan-out point; it is advisory only, and the server does not implement `Last-Event-ID` replay (gap recovery is re-hydration by design, a conscious keep per `PORT-READINESS.md`). Drop-oldest is correct only because of push-to-pull (the newest frame triggers the freshest hydration); a backpressure or drop-newest channel is wrong here. Pinned by: `test_event_stream.py` (drop-oldest, fan-out reach); the literal 256, the starting value, and cross-connection monotonicity are unpinned (V-4).

- **E-8 [M/I] Thread boundary and shutdown order.** Serialisation happens on the publisher's thread; the hop to the async runtime happens in exactly one place and one direction; the connection registry and sequence counter are touched only on the runtime side. On shutdown the hub detaches first, before producers stop, and a post-shutdown publish is a safe no-op (dropped frame), never a panic. Pinned by: `test_event_stream.py` (cross-thread delivery, close-unsubscribes); the supervised-workers runtime checks.

### Time

- **T-1 [M/B] Every time read flows through the injected clock; golden-observed event timestamps derive from the domain instant.** The seam is `backend/testing/clock.py` (wall and monotonic streams advance in lockstep), injected at construction into every service and reached by the router layer through the service container; the no-ambient-time guard (`backend/scripts/check_ambient_time.py`) holds the production packages at zero ambient reads. Replay scenarios drive a deterministic clock from the committed per-scenario clock plan (`backend/testing/clock_plan.py`: frozen at the plan's start, advanced only by the replay driver, never by reads), which is what makes the `<TS_N>`-normalised goldens reproducible and read-count-independent across implementations. Stamping an instant that is not plan-derived into a golden-observed output (the tracking event's `occurred_at`, a persisted timestamp column, a response field) breaks the symbol mapping and fails the goldens. The scan status event still stamps its publish instant (the scan flow has no domain instant), but the read goes through the injected clock like every other; the tracker's timestamp-absent fallbacks likewise synthesise from the injected clock. One deliberate carve-out: the chat-log watcher's drain-deadline monotonic reads go through its own injected clock, which deterministic harnesses leave real so a failing drain times out rather than hanging. Timestamps parsed from the chat log are naive local-time instants converted through the host timezone, and clock-plan starts share that naive-local interpretation; reproduce it rather than assuming UTC parsing (V-4). Pinned by: the fingerprint and DB-state goldens (every observed timestamp normalises to a scenario-derived symbol); `test_clock.py` for the seam contract; `test_ambient_time_guard.py` for the zero-ambient-reads scan.

### Concurrency and actors

- **C-1 [M/I] Actor translation: one owner per mutable state, events published only after the synchronisation region ends.** Each actor-shaped service (`HuntTracker`, `SkillScanManual`, the skill tracker, the listeners) becomes one struct owning its state; whether the port keeps a lock or moves to a single-owner task, no subscriber may ever run while the producer's synchronisation region is held, and no service holds its region across a bus publish or a database call. The borrow checker verifies a design that already exists; do not invent topology the Python design lacks. Pinned by: event ordering in the fingerprint goldens; `test_tracker_concurrency.py` (torn-read hammer).

- **C-2 [M/B] Bus dispatch is synchronous and inline on the publisher's thread; subscribers are error-isolated.** A publish completes its subscribers' work before returning: that synchrony is what makes "per-event publishes, then the tick boundary, then the coalesced update" a deterministic recorded order. Deferring handler work to a queue reorders the recorded stream and breaks every fingerprint. A failing subscriber is logged and skipped; the rest still run. Pinned by: the fingerprint goldens (the order is the synchronous dispatch order).

- **C-3 [M/B] The tracker's outbound notification is dirty-flag coalesced: one `tracking.session.updated` per settled tick, and only when a tracker-owned mutation happened.** Mutating handlers set the flag only on a genuine mutation (a deduplicated heal, an unhandled event type, an ignored enhancer break must not set it); the tick boundary handler reads-and-resets under the lock and publishes after release, stamping the tick's timestamp and the session id captured under the lock. A tick carrying only events other services own (a bare skill gain) produces no update. Reasons: `started` on start, `updated` on a dirty tick, `stopped` on stop; the start and stop frames stamp the session's start and end instants rather than a tick timestamp. Pinned by: the `skill_gain_across_tick` scenario goldens (the skill-only tick is the sharpest negative).

- **C-4 [M/B] Within-tick publication order is fixed, loot is one grouped frame, and the tick boundary is published last.** Per-event frames flush in the watcher's fixed partition order, then `tick_flushed` after all of them; the loot group is a single frame with `items[]` and a 4dp-rounded total, never per-item frames. The coalesced session update therefore always follows the tick boundary in the recorded stream. Pinned by: the fingerprint goldens (order-sensitive by construction).

- **C-5 [M/B] The scan service synthesises its settled boundary: compare-and-advance on a projected status key, baselined to idle at construction.** The status key covers only the service-owned fields; the comparison and advance happen under the lock, the publish after release; the resting state is silent (the idle baseline), each verb completion and per-page step emits exactly one frame on a genuine key change; the payload is the coarse phase only. Pinned by: `test_skill_scan_producer.py` (one frame per discrete change, resting silence).

- **C-6 [M/B] The chat-log watcher is one owned task: seek to end, signal ready, tail at 100 ms; unparsed lines never close a tick.** `start` blocks until the seek-to-end has run so a write issued immediately after cannot be missed; the 100 ms cadence is part of the recorded-scenario timing model and stays even if the discovery mechanism changes (`PORT-READINESS.md`); lines that parse to nothing must not flush the pending tick, because unrelated chat traffic shares timestamps with events that belong together. The watcher is the single producer of all chat-driven events. Pinned by: the replay corpus end to end (which depends on the tail's timing shape); `test_supervised_workers.py` (ownership, termination); the literal cadence value itself carries no direct assertion.

- **C-7 [M/I] Subscriptions are session-scoped: an idle tracker consumes nothing.** The tracker subscribes its low-level handlers on session start and unsubscribes on stop, under the lock, with the dirty flag reset ordered alongside; starting a new session stops any prior one first so its stop events publish cleanly. In a single-owner port this may become an enabled flag on the owner; the observable (an idle tracker emits nothing and mutates nothing) is what must hold. Pinned by: the `empty_session` scenario golden (start and stop frames only).

- **C-8 [M/B] Stateful one-shot suppressions are exact: arm, consume on first match, expire by clock, clear at both session boundaries.** The skill tracker's claim suppression arms a per-skill entry with a 30-second expiry, consumes it on the first matching gain whether or not the window has lapsed (delete before the window check), and clears all armed entries on both session start and stop. The watcher's quest-reward filter may suppress one loot item and/or one skill gain (at most one of each, independently) from the tick that completes a mission. Pinned by: the `mission_completion_with_reward_suppression` scenario goldens.

- **C-9 [M/B] The keystroke seam and the allowlist survive the port.** Production code consumes a keystroke-source abstraction, never the hook library; the mock twin delivers injected events only while running (a pre-start inject is dropped); the constructor allowlist filters at the hook boundary, and the production allowlists are exactly the digit keys for the hotbar listener and the space key for the capture listener. These are the input-minimisation posture and the replay substrate; both are non-negotiable. Pinned by: `test_keystroke_source.py`, `test_hotbar_listener.py`, and `backend/tests/e2e/test_input_listening_minimisation.py` (recorded keystrokes lie within the union of the two allowlists).

- **C-10 [M/I] Worker discipline carries over: every long-lived worker is named, owned by a service, and joined on shutdown; no free-floating tasks.** The static and runtime enforcement of `test_supervised_workers.py` is the contract; the port's equivalent is owned, named tasks with deterministic shutdown, and the SSE fan-out continues to own no background task (each connection's drain belongs to its request lifecycle). Pinned by: `test_supervised_workers.py` (the discipline), adapted to the port's runtime.

### Numerics

- **N-1 [M/B] Python `round(x, n)` is round-half-to-even on the IEEE-754 binary value; reproduce it exactly, never with round-half-away.** `round(0.125, 2) == 0.12`, `round(2.675, 2) == 2.67` (2.675 is stored below the literal), `round(0.135, 2) == 0.14`. A naive scale-round-unscale with half-away ties, or `f64::round`, diverges on exactly the tie cases the mutation suites pin. Single-argument `round(x)` returns an integer with the same tie rule; `int(x)` truncates toward zero. Integer `%` in the ported scope is proven non-negative (codex ranks, level counts); do not extend Python modulo semantics assumptions to negative operands. Pinned by: exact-value pins in `test_mut_cost_engine.py`, `test_mut_tt_value_curve.py`, the codex/character suites; the Rust-side helper carries obligation V-1.

- **N-2 [M/B] The rounding precision map is per-field and heterogeneous; never unify it.** The same response mixes 1dp (skill levels), 2dp (costs, returns, net, PED amounts), 4dp (return rates, TT values, reward PED, timeline buckets), and 6dp (codex weighting figures); the cost engine works at 4dp, session aggregates at 2dp with 4dp rates. Carry each emit point's exact precision from the source (`backend/services/`, `backend/routers/`); a uniform precision is wrong by construction. Pinned by: the mutation and property suites per module; the 4dp cases additionally by the normalised goldens.

- **N-3 [M/B] Accumulation order is part of the arithmetic: round per line, then round the total.** The cost engine rounds each line's figures to 4dp, sums the already-rounded values, and rounds the total again; summing raw products and rounding once can differ in the fourth decimal place. Preserve the order of rounding relative to summation wherever it appears. Pinned by: `test_mut_cost_engine.py` exact pins.

- **N-4 [M/B] Falsy coalescing, not null coalescing.** Python `x or default` substitutes the default for `None`, `0`, `0.0`, and `""` alike, and `if not x` matches the same set; the leaf services lean on this (a zero decay, a zero uses-per-minute, a zero markup all fall back). A null-only `unwrap_or` keeps the zero the Python code replaces. Reproduce or-falsy semantics case by case. Pinned by: `test_mut_cost_engine.py` (the falsy-fallback pins).

- **N-5 [M/B] `bisect_right` translates to `partition_point` with a less-than-or-equal predicate; binary search by equality is forbidden.** The curve lookup is clamp-then-`bisect_right`-minus-one with linear interpolation at 4dp; the exact equivalent is `slice::partition_point(|&e| e <= x) - 1` under the existing clamp guards. A by-equality binary search returns an arbitrary index among equal elements and breaks the anchor selection. The level-for-value inverse, in full: a non-positive amount returns `0.0` immediately; a target value at or above the curve's top returns the top level minus the starting level (greater-or-equal, not greater); otherwise exactly 64 iterations of midpoint bisection seeded with the starting level as the lower bound and the curve's top level as the upper, stepping `mid = (lo + hi) / 2` and assigning `lo = mid` when the value at `mid` is strictly less than the target, else `hi = mid`; the result is the lower bound minus the starting level, rounded to 4dp. The iteration count, the strict less-than, the branch assignment, and returning the lower bound are all load-bearing. Pinned by: `test_mut_tt_value_curve.py` (boundary and 4dp pins, the 64-iteration outputs).

- **N-6 [M/B] Constant tables port verbatim, values and order.** Reward divisors, the codex multiplier ladder, the damage-type ordering, the attribute-skill set, the profession divisor, base HP: fixed constants with observable consequences. Port them as constants with identical values and ordering. Pinned by: the codex/character formula suites.

- **N-7 [M/B] Fixed-precision format strings keep trailing zeros and round half-to-even.** The f-string forms (`{:.1f}`, `{:.2f}`) produce `100.00`, `0.1` for `0.05`, `0.12` for `0.125`: trailing zeros preserved, ties to even on the binary value. They feed string fields (the trifecta range, notable-event descriptions) that no corpus scenario currently captures, so the port adds its own pins (V-2). Pinned by: none today.

- **N-8 [J/B] Shortest-round-trip float rendering matters on the fields that reach the wire unrounded.** Most numeric fields are rounded before serialisation, insulating them from renderer differences; the trifecta description fields (per-shot and per-use PED costs, damage and heal ranges) and the weapon damage profile carry raw `f64` values to the wire, where Python's `repr` and Rust's shortest-round-trip rendering must agree. The 4dp golden normalisation masks sub-4dp drift, so this is a byte-level obligation (V-3), not a golden-graded one. Pinned by: none end-to-end today (property suites are tolerance-based).

### Persistence

- **P-1 [M/B] The connection opens with exactly four pragmas: `journal_mode = WAL`, `synchronous = NORMAL`, `busy_timeout = 5000`, `cache_size = -8000`.** WAL is an on-disk observable (header magic, `-wal`/`-shm` sidecars); the others govern durability and latency behaviour. No golden reads pragma state, so the port verifies by pragma readback (V-7). Source: `backend/db/base.py`.

- **P-2 [M/B] The fresh-install schema reproduces the current `sqlite_master` verbatim: every table, index, trigger, and default, including `unixepoch('now')` defaults and the `trg_fill_*` backfill triggers.** The schema sources are `backend/db/app_database.py` and `backend/tracking/schema.py`; acceptance is identical normalised `sqlite_master` text between a fresh Python database and a fresh ported one. The timestamp-backfill triggers (fill-on-NULL with the engine clock) are reproduced for schema fidelity even though their filled values never reach a golden (P-4). Pinned by: the six-table catalogue for observed columns; the full schema text needs the port-side equivalence assertion (V-8).

- **P-3 [M/B] The schema-version row is part of the user's database.** `db_metadata(key, value)` stores the version as a stringified integer under `version`; a fresh database reads 0, a current one stores the live version; migration runs only when current is below target. Whatever migration machinery the port adopts must read and honour this row on existing user databases. Pinned by: none in the corpus (replays start at schema head); V-9 covers the upgrade test.

- **P-4 [M/B] Every golden-captured column is app-supplied through the injected clock; engine-defaulted timestamp columns never reach captured output.** Session boundaries, kill timestamps, loot rows, ledger dates: all bound parameters stamped by the application. The `unixepoch('now')`-defaulted columns (`created_at`, `updated_at`, and friends) are wall-clock and are deliberately excluded from the snapshot catalogue. The port must not let an engine-defaulted value flow into any observed surface, and must not "simplify" a default away (P-2). Pinned by: `db_state.json` per scenario over the explicit-column catalogue.

- **P-5 [M/B] Snapshot determinism: explicit column lists, the catalogue's per-table ordering taken verbatim, COALESCE shapes, and insertion order preserved.** The capture queries select named columns (never `*`) with per-table deterministic ordering that is heterogeneous and enumerated by the catalogue (`backend/testing/db_snapshot.py`): `tracking_sessions` and `ledger_entries` order by `rowid` alone; `kills` and `notable_events` by `timestamp` then `rowid`; `kill_loot_items` and `kill_tool_stats` by the joined parent kill's `timestamp` then their own `rowid`. Take each table's ordering from the catalogue, not from a generalisation; and never order by a generated id (UUIDs symbolise after the SQL sort, so id-ordering flips across runs). `tracking_sessions` captures `COALESCE(heal_cost, 0.0)` and `COALESCE(dangling_cost, 0.0)`: the golden expects float `0.0` where the column is NULL. Rowid order reproduces only if the port inserts rows in the same chat-stream order the Python tracker does. Pinned by: `db_state.json` (the capture encodes all of it).

- **P-6 [M/B] Write-unit semantics: same commit boundaries, same statement verbs.** The tracker commits once after the session-start insert and once per kill's full write unit (the kill row plus its tool stats and loot items together); the kill row and tool stats use INSERT OR REPLACE while loot items use plain INSERT (re-finalisation replaces the parent and appends children); the version row commits immediately. Mid-run snapshots (the consistency midpoints) observe committed state, so coarser batching is observable. Pinned by: the midpoint-consistency goldens and `db_state.json`.

- **P-7 [M/B] The in-place upgrade ladder is behaviour, not history.** Migration from a pre-floor version refuses loudly (it must never stamp the current version over an unmigrated schema); the forward steps are defensively idempotent (missing-table and duplicate-column conditions are caught and logged, not fatal). The corpus never exercises upgrades, so the port writes its own test: upgrade a seeded old-version database and assert the result matches the fresh-install schema (V-9). The Python code matches SQLite error text by substring; the port's database binding may surface errors differently (V-9). Source: `backend/db/app_database.py`.

### Files and configuration

- **F-1 [M/B] `settings.json` is a byte-stable format per platform: two-space indent, ASCII-escaped, keys in existing-then-declaration order.** A save merges the on-disk document under the typed config's fields (existing keys first, then the config's fields in declaration order, which overwrite). Two byte-form facts differ from the golden writers and are easy to get wrong: non-ASCII characters are escaped (`\uXXXX`), not written as multibyte UTF-8, and line endings are platform-native (carriage-return line-feed on Windows) because the save path uses text-mode newline translation. Round-tripping an unchanged config is a byte-identical fixed point on a given platform. Pinned by: the config property suite (`test_config_service_properties.py`, the fixed-point test; its inputs are ASCII-only and same-platform, so the two byte-form facts are code-sourced rather than test-pinned).

- **F-2 [M/B] Saves are atomic: write the temp file, best-effort rename the current file to `.bak`, rename temp into place.** The two renames happen in that order; the backup rename's failure is suppressed (best effort) and never aborts the save; a crash between the renames leaves either the old file or the new file plus a backup, never a truncated `settings.json`. A missing or unparseable file on load falls back to defaults and immediately saves them. Pinned by: `test_config_service.py` (backup exists after overwrite; corrupt file replaced by defaults); the crash ordering itself is untested, so preserve it by construction.

- **F-3 [M/B] Unknown keys: dropped when applying updates, carried forward on disk.** Updates to the typed config silently ignore unknown field names; a save preserves any unknown keys already persisted in the file (third-party tooling's values survive a save by a process that does not know them). A strict struct round-trip loses them; the port needs an explicit catch-all field (register entry D-5) and a positive survival test (V-9). Pinned by: the in-memory drop is tested; the on-disk survival currently is not.

- **F-4 [M/B] The game-data snapshot is read-once, in-memory, with its quirks intact.** Every `backend/data/snapshot/*.json` loads once at startup keyed by file stem; `skill_ranks` is the single-object exception (wrapped into a one-element list); a parsed payload of the wrong shape (an object where a list is expected, outside the single-object exception) degrades to an empty list with a warning, while an unparseable file raises at startup; search is case-insensitive substring over the display name (`species.name` for mobs, `name` otherwise); id lookup compares stringified ids. Pinned by: the HTTP goldens of the endpoints served from the store (indirect).

### Golden and corpus discipline

- **G-1 [M/B] "Byte-faithful" means byte-equal after the shared normalisation, which masks exactly three things.** The corpus normaliser (`backend/testing/fingerprint.py`) replaces UUID-shaped strings with `<UUID_N>`, timestamps (datetimes, ISO-prefixed strings, bare floats in the epoch window 1e9..2e10) with `<TS_N>`, and rounds other floats to 4dp. Everything else is observed verbatim: key sets, string content, exact integers, list order, and the None/bool/int distinction (bool dispatches before int, so a boolean never serialises as a number; a port that flips a bool to an int diverges). A genuine value that strays into the epoch float window would be mis-symbolised as a timestamp; keep observable values out of that window (V-11). Source: `backend/testing/fingerprint.py`.

- **G-2 [M/B] Symbols are assigned by first encounter across the whole scenario, through one normaliser shared by all three surfaces.** `<UUID_1>` is whichever UUID surfaced first across the event stream, the database snapshot, and the HTTP bodies, in that capture order; the tables reset per scenario. Reproducing a golden therefore requires the same publish order and the same first-encounter order of identifiers; an emission reorder renumbers symbols everywhere downstream and produces a large diff that looks like a data change but is an ordering change. Source: `backend/testing/fingerprint.py`, `backend/testing/db_snapshot.py`, `backend/testing/http_fingerprint.py`.

- **G-3 [M/B] The HTTP fingerprint projects a narrow, fixed axis set; reproduce the axes, enjoy latitude elsewhere.** Status code; the three lowercased headers `content-type`, `cache-control`, `etag` (a strong ETag projects to a sentinel, anything else stays verbatim and surfaces as a diff); the normalised JSON body (a numeric value under a key named exactly `duration` masks to a sentinel, because a session's wall-clock length reflects replay latency, not contract); non-JSON bodies project to a binary marker with byte length. Headers outside the three are free latitude for this oracle (but see M-6 and V-5). Source: `backend/testing/http_fingerprint.py`.

- **G-4 [M/B] Golden regeneration is canonical text.** Sorted keys, two-space indent for the JSON documents, one compact line per event for the JSONL, UTF-8, LF endings, trailing newline. Any port-side regenerator matches this byte form. Source: the writers in `backend/testing/fingerprint.py`, `db_snapshot.py`, `http_fingerprint.py`.

- **G-5 [J/B] Goldens never regenerate to make a port pass, and the normalisation mask never widens for one.** A failed golden means a divergence to fix or an intended behaviour change to take through the documented regeneration discipline (diff surfaced, the regeneration commit marker, the independent ratification record; `TESTING.md`). Adding a mask, sentinel, or ignore entry to absorb a port mismatch removes a surface from observation: it is a harness change with the same review bar as a golden move, justified only by genuine nondeterminism. First pins (a golden that did not exist before) deserve the most scrutiny, because nothing fails when they are wrong. Pinned by: the golden-ratification CI guard for golden moves; review discipline for mask changes.

## Deliberate-divergence register

Each entry: the change (or, in one marked case, a conscious keep recorded so it is not "fixed" in passing), why it is an improvement, and the oracle asset that still proves equivalence despite it. The register is closed; a new divergence is added here (with its proving asset) before it is implemented, not after.

- **D-1 The database gets one owner.** The shared connection with an unevenly-honoured lock, the one-directional lock-order invariant, and the provider-callback exception (`PORT-READINESS.md`, "The shared SQLite connection") do not translate; the port routes every service through a single-owner writer with explicit transaction scope. Equivalence: the per-scenario `db_state.json` and fingerprint goldens observe only committed state, so the locking strategy is invisible to them as long as commit boundaries (P-6) and insertion order (P-5) are preserved.

- **D-2 The domain-event channel is monomorphic.** A typed channel carrying only the domain-event union replaces the untyped bus convention, making "a typed instance on a domain topic" a compile-time guarantee; the SSE hub's runtime type-guard branches become dead code and are omitted (their guarded path, a malformed payload, is never observable on the wire). Equivalence: `event_schemas.snapshot.json`, the wire-dict assertions, and the SSE seam tests pin that the frames crossing the wire are unchanged.

- **D-3 Spawn-per-keypress becomes one owned worker.** The short-lived per-press threads exist only because a thread was Python's cheapest way to get off the OS-hook callback; the port's hook callback enqueues onto a channel drained by one owned worker. Equivalence: the listener test suite (off-hook dispatch, press-edges only, allowlist) and the recorded bus-event pins for tool changes; no oracle observes the worker mechanism.

- **D-4 Hand-rolled FFI becomes platform-crate bindings.** The window enumeration, process-priority, and keyboard-hook glue move from `ctypes` calls to the platform crate equivalents; packaging detection is redesigned for the new build pipeline. Equivalence: none of the FFI mechanics are load-bearing architecture (`PORT-READINESS.md`); the keystroke seam tests, the scan-geometry suites, and end-to-end liveness pin the behaviour that matters.

- **D-5 Config persistence gains an explicit catch-all.** Python's unknown-key carry-forward is an implicit dict merge in the save path; the port expresses it as a typed catch-all field for unrecognised keys, making the contract visible and type-checked. Equivalence: the config round-trip fixed-point test pins the known-key byte shape; a new positive test pins unknown-key survival (V-9).

- **D-6 (resolved on the Python side) The clock seam is total on both sides.** Originally a port-side divergence: a handful of ambient time reads survived in Python (timestamp-absent fallbacks, the scan status event's publish-time stamp). The Python side has since been completed ahead of the port: every wall-clock and monotonic read in the production packages now flows through the injected `backend/testing/clock.py` clock, constructed once at the composition root, and the no-ambient-time guard (`backend/scripts/check_ambient_time.py`) holds the tree at zero ambient reads. The port reproduces the same shape directly, and replay scenarios drive both implementations from the same committed per-scenario clock plan (`backend/testing/clock_plan.py`), which is what makes timestamp-bearing outputs directly comparable. Equivalence: unchanged goldens under the plans, the seam contract in `test_clock.py`, and the guard's zero-site scan.

- **D-7 (conscious keep) Gap recovery stays push-to-pull.** Not a change but a conscious keep, recorded so it is not "fixed" in passing: the SSE sequence id stays advisory, `Last-Event-ID` replay stays unimplemented, and reconnect recovery stays re-hydration plus the relay's nudge (`PORT-READINESS.md`, "Known hazards"). Equivalence: the seam tests and the stream contract (E-6, E-7).

## Verification obligations

Empirical checks the port owes before relying on the adjacent rules blind. Each discharges into either a port-side test or a recorded probe result; an obligation left open is a known blind spot, not a permission.

- **V-1 (rounding parity).** Prove the port's rounding helper reproduces CPython `round()` on the pinned tie cases at 2dp and 4dp (`0.125 -> 0.12`, `0.135 -> 0.14`, `2.675 -> 2.67`, the curve pins `tt_value_at(100.5) == 0.125` and `levels_for_tt_value(0.0, 17.68) == 2446.3317`), and probe a live fingerprint corpus replay before the first arithmetic-bearing service relies on it. (Rule N-1.)
- **V-2 (format strings).** Pin the fixed-precision string formats port-side (`{:.1f}`/`{:.2f}` ties and trailing zeros); no corpus scenario captures them today. (Rule N-7.)
- **V-3 (wire bytes).** Confirm the port's JSON serialisation matches the live Python wire form byte for byte on representative bodies: compact separators, insertion-order keys, trailing `.0` on integral floats, shortest-round-trip rendering on the unrounded fields (N-8), UTF-8 without ASCII escaping. The ETag (a body-bytes hash) is the in-production observable; a real conditional-request round-trip across implementations is the convincing check. (Rules W-7, M-1.)
- **V-4 (unpinned stream details).** Add port-side pins for the SSE keep-alive text and cadence, the `Connection` and `X-Accel-Buffering` headers, the queue capacity 256, the sequence start at 1, the fractional `occurred_at` rendering, the payload's always-present nullable `sessionId` key, the envelope key serialisation order, and the timezone interpretation of chat-log instants (parsed as naive local time and converted through the host timezone, an interpretation the per-scenario clock-plan starts share); all are code-sourced today with no test assertion. (Rules E-2, E-3, E-6, E-7, T-1.)
- **V-5 (CORS and guard bodies).** Verify CORS behaviour (preflight included) and the three 403 detail bodies empirically against a real webview origin, and pin them port-side; the fingerprint projection deliberately drops both. (Rules M-5, M-6.)
- **V-6 (error-to-status sweep).** Sweep every router's error paths (status code plus detail string) into a port-side table and pin them; the corpus pins 200 paths only. The integer-id overflow case (an out-of-range path id raising a 500 today) is reproduced faithfully, not fixed in passing. (Rule W-6.)
- **V-7 (pragma readback).** Assert the four pragma values by readback on the port's connection; no golden observes them. (Rule P-1.)
- **V-8 (schema text equivalence).** Author the `sqlite_master` comparison between a fresh Python database and a fresh ported one; the snapshot catalogue grades six tables' columns, not the schema text. (Rule P-2.)
- **V-9 (persistence and config edges).** Write the in-place upgrade test (seeded old-version database upgraded equals fresh install); confirm the database binding's error surface against the substring matching the defensive migrations rely on; add the positive unknown-key survival test for `settings.json`. (Rules P-3, P-7, F-3.)
- **V-10 (wire-layer serialisation strategy).** Decide, at the wire-layer port, how the mixed-casing, open-model, absent-vs-null contract is expressed (per-field renames on structs, handler-built value trees, or a custom serialisation layer), against the constraints of W-2, W-4, and W-5 together. The constraint set, not idiom preference, decides. (Rules W-2, W-4, W-5.)
- **V-11 (epoch-window audit).** Check no genuine observable value (monetary totals, counters) can land in the normaliser's epoch float window; a new scenario or surface that emits one would be silently mis-symbolised. (Rule G-1.)
- **V-12 (conformance re-host).** Re-point the schemathesis contract suite at the ported server early (the first hydration routes are enough) to learn whether the re-host is purely mechanical (host and port) or needs per-route adjustment; record the answer beside the suite.

## Growth protocol

This rulebook grows by appendix, not by rewrite. When a port surfaces a rule this document missed (a fix-loop discovery, a review finding, a divergence that needed adjudication), add it: in the right section, with the next free identifier, a citation, the oracle that grades it, and a MECHANICAL or JUDGEMENT tag. Identifiers are stable and never reused; superseded rules are marked superseded in place rather than deleted. Two hard edges: a new entry in the divergence register requires its equivalence-proving asset named at the time of entry, and nothing in this document can widen what the corpus normalisation masks (that is a harness change under G-5's bar, not a rulebook edit).
