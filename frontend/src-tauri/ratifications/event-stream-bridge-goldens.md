## Set 2: crit_dodge_evade_jam fingerprint and db_state

### Stated rationale and the independent db_state question

The author's rationale covers `fingerprint.jsonl` (the scenario's event stream
gains the new `tracking.session.updated` envelope frames the event-stream
bridge introduced; pre-existing frames otherwise unchanged) but supplies no
rationale for the 8-line `db_state.json` delta; it was judged independently and
is a benign consequence of the same commit, explained below.

### The canonicalisation mechanism (key to both files)

Goldens are canonicalised by a single shared `Normalizer`
(`backend/testing/fingerprint.py`): `<TS_N>` symbols are assigned in
first-encounter order across the shared lifetime spanning the fingerprint event
stream and the DB snapshot. New distinct timestamps that appear earlier in the
stream therefore renumber every later symbol. This is the lens through which the
db_state delta resolves.

### fingerprint.jsonl: delta accountability

The scenario's `chat_replay.log` has four distinct chat timestamps (four settled
ticks: `10:00:00`, `:01`, `:02`, `:03`), each of which mutates the live session.
The final fingerprint adds exactly:

- 1 `tracking.session.updated` with `reason: "started"` (session start),
  `<TS_1>`.
- 4 `tracking.session.updated` with `reason: "updated"`, one per settled
  mutating tick, each paired with one `tick_flushed` frame, stamped with that
  tick's own domain timestamp.
- 1 `tracking.session.updated` with `reason: "stopped"` (session stop),
  `<TS_10>`.

Every pre-existing frame (session_started, the seven combat events, loot_group,
session_stopped) is preserved with identical payload values (damage
14.0/17.0/30.0/12.0, quantity 540, value_ped 5.4, total_ped 5.4) and identical
relative ordering; only their `<TS_N>` symbols renumber because the new frames
interleave new distinct timestamps. Nothing pre-existing was reordered,
revalued, or dropped.

### Frequency scrutiny (the failure mode this review exists to catch)

The specific risk is a new event firing on no-op ticks and that over-emission
being pinned as truth. It is not present here:

- The emitter `HuntTracker._on_tick_flushed` returns early on
  `self._session is None or not self._session_dirty`, so it fires only on a
  tick that actually mutated the session readout; the combat/loot handlers set
  the dirty flag, an unhandled or duplicate event does not.
- The count is exactly one `updated` frame per settled mutating tick (4 ticks,
  4 `tick_flushed`, 4 `updated`: strictly 1:1), plus the two session-boundary
  frames. Not per-event, not per-line.
- Decisive cross-check: the sibling `empty_session` scenario (zero mutating
  ticks) gains exactly two frames, `started` and `stopped`, with **no**
  `updated` and **no** `tick_flushed`. A no-op session emits no per-tick
  frames. Had the emission been mis-gated, that scenario would carry spurious
  `updated` frames; it carries none. The sibling `single_mob_hunt` shows the
  identical per-mutating-tick structure, confirming a uniform corpus-wide
  treatment rather than an isolated anomaly.

### Determinism

The new envelope's `occurred_at` is stamped from the domain timestamp that
triggered the event (the tick time, or the injected session start/stop
instant), never a fresh clock read. `to_iso_utc` is a pure function of the
passed timestamp (`datetime.fromtimestamp(ts, tz=UTC)`), and session-boundary
instants flow through the injected `now_fn` seam the replay harness drives
deterministically. The `occurred_at` symbol for a tick differs from that tick's
combat-event symbol (e.g. `<TS_3>` vs `<TS_2>`) only because the envelope
serialises the instant in UTC ISO form while the combat events carry the naive
local ISO form: a deterministic encoding difference yielding a distinct raw
string, not ambient input. No wall-clock, randomness, environment, or
machine-timing value leaks into the pinned output.

### db_state.json: independent judgement

The full before/after of `db_state.json` is byte-identical except for four
timestamp placeholder symbols, each shifted by a constant `+6`:
`<TS_5> -> <TS_11>` (`ended_at`), `<TS_6> -> <TS_12>` (`started_at`),
`<TS_7> -> <TS_13>` (kill `timestamp`), `<TS_8> -> <TS_14>` (ledger `date`).
Every concrete value (shots_fired 7, damage_dealt 73.0, critical_hits 1, ledger
amount 0.054, quantity 540, value_ped 5.4, all UUIDs, descriptions, tags, and
flags) is unchanged. The `+6` offset is exactly the number of new distinct
timestamps the fingerprint stream now encounters ahead of the DB columns under
the shared `Normalizer` (the session-start instant plus the per-tick
`occurred_at`/UTC instants). The DB rows themselves did not move; only their
canonical symbols renumbered as a mechanical consequence of the new upstream
emissions sharing the symbol table. No DB column gained, lost, or changed a
real value. This is a benign symbol renumbering, not a state regression.

### Precedent

`backend/testing/ratifications/hydration-only-tracking-surface.md` ratified the
equivalent deltas for the other scripted scenarios in a later commit; the
structure confirmed here (additive event frames, symbol renumbering only,
surviving content untouched) is consistent with that ratified precedent.

## Findings

- **Coverage matrix: a genuine, additive spec move.** The net matrix delta is
  three additive entries (one new `event_stream` row with Services `24 -> 25`;
  two test-name additions to existing rows), each mapped to a verified
  service-file or test-file addition in the same commit. No coverage removed
  or reclassified.
- **Fingerprint delta fully accounted for.** Added frames are exactly the
  session-boundary and per-settled-tick `tracking.session.updated` envelopes
  plus their `tick_flushed` companions; all pre-existing frames preserved in
  value and order, symbols renumbered only by interleaving.
- **Frequency contract verified.** Emission is strictly 1:1 with settled
  mutating ticks (dirty-gated), corroborated by the zero-tick `empty_session`
  sibling emitting no per-tick frames. The no-op-tick over-emission failure
  mode is absent.
- **Determinism clean.** `occurred_at` is a pure transform of
  replayed/injected domain timestamps via `to_iso_utc` and the `now_fn` seam;
  no ambient input enters the pin.
- **DB snapshot: benign renumbering.** The initially unexplained 8-line delta
  is a constant `+6` renumbering of four timestamp symbols under the shared
  canonicaliser, with every concrete value byte-identical. A benign
  consequence of the new upstream emissions, not a state change.

Nothing suggested a swept regression or a nondeterministic pin; nothing
required escalation.

## Summary judgement

Both deltas are faithful, minimal projections of intended behaviour changes,
not regressions laundered into the goldens. The coverage matrix gained three
derived entries that each trace to a real service or test added in the same
commit, with nothing removed or reclassified. The `crit_dodge_evade_jam`
fingerprint gained exactly the event-stream bridge's session-boundary and
per-settled-tick envelopes at the correct once-per-mutating-tick frequency (the
central over-emission failure mode is provably absent, confirmed against the
zero-tick sibling), with every pre-existing frame and value preserved; its
`db_state.json` delta is a pure timestamp-symbol renumbering with no concrete
value moved. The new output is deterministic, stamped from replayed and
injected instants through a pure transform. Adapting the oracle to include
these emissions is the correct action; the prior pins were genuinely obsolete.
